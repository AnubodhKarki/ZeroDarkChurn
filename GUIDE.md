## The Problem This Solves

Imagine you run a developer-facing API company — in this case DevAPICo, a speech-to-text API. You have 100 paying customers. Some of them are quietly drifting away. They are not filing cancellation tickets. They are not complaining loudly. They are just making fewer API calls each week, running into more errors, and privately exploring competitors.

By the time a customer formally churns, it is usually too late to save them. The warning signs were there weeks earlier — a support ticket saying "we're evaluating alternatives," API error rates creeping up, call volume dropping 40% week-over-week — but no one connected the dots in time.

This is the "silent churn" problem. The goal of this system is to detect it early, automatically, by correlating quantitative health metrics with qualitative communication signals, and then automatically drafting a personalised outreach email for the Customer Success team to review and send.

---

## Why This Is Architecturally Interesting

Most simple "AI" projects are just: take some text, pass it to GPT, show the output. That is not what this system does.

The system has three distinct layers that each do something different:

1. **A pure statistics layer** (no AI) — detects whether something has changed in the numbers
2. **A language model layer** — reads human communication and extracts what the customer is actually thinking
3. **A business-rules layer** — combines the two signals with account value to decide whether to act

Each layer is independently testable and independently replaceable. The system is designed so that a bad LLM response cannot cause a false outreach — the business rules layer acts as a final gate.

## The Data

Before explaining the pipeline, it helps to understand the data it operates on.

### What data exists per account

For each of the 100 synthetic accounts, there are five data sources:

**`accounts.csv`** — One row per company. Fields include: account_id, company_name, tier (free / starter / growth / enterprise), funding_stage (seed / series_a / series_b / series_c), current_mrr_usd, primary_use_case.

**`usage.csv`** — 90 rows per account (one per day). Fields include: api_calls, error_count, error_rate, webhook_attempts, webhook_success_rate, rate_limit_hits_429, unique_endpoints. This is the time-series signal — the heartbeat of the customer relationship.

**`tickets.json`** — Support tickets with subject, body, status, created_at. These are the clearest signals of friction. A ticket saying "your API keeps returning 503 and our investors are asking about the downtime" is a very different ticket from "how do I set the language parameter."

**`slack.json`** — Internal Slack messages from the customer's team (in a shared channel). More informal and candid than tickets. This is where you find things like "ping me when the contract renewal comes up, I'm not sure we're getting enough value."

**`transcripts.json`** — Call transcripts from customer success calls. The richest source of signal — full conversations, often with explicit mentions of pain points, timelines, budget discussions, or competitor names.

### How the data was generated

All data is synthetic, generated by `data/generate.py`. The generator uses an LLM to write realistic tickets, Slack messages, and transcripts. For the 80 ordinary accounts, it generates normal-sounding communications. For the 20 "planted" accounts, it injects specific language.

The **15 planted churn accounts** fall into three groups:
- `acc_0001`–`acc_0005`: **health_only** — usage metrics collapse (big drop, error spike, webhook failures) but their communications sound neutral. These accounts are churning quietly — the numbers tell the story, the words do not.
- `acc_0006`–`acc_0010`: **risk_only** — communications explicitly mention competitors (fictional names: VoxCore, TranscriBit, SpeechLayer, AudioNest), budget concerns, or cancellation intent. But usage is only mildly dipping. The LLM must catch these.
- `acc_0011`–`acc_0015`: **combined** — both signals are present. Easiest to detect.

The **5 false-positive traps** are designed to look like churn but not be:
- `acc_0016`–`acc_0017`: **seasonal dip** — volume drops for 7 days in a row, triggering the week-over-week detector, but it is explained in the communications as a planned holiday or campaign lull. The LLM reads this context and correctly says "low risk."
- `acc_0018`–`acc_0019`: **transient error spike** — error rate shoots up briefly (a deployment issue they fixed themselves). Single-signal anomaly. No usage drop. Communications are positive.
- `acc_0020`: **loyal competitor mention** — they mention a competitor by name ("we looked at VoxCore but decided to stay with you") but in a positive context.

The remaining 80 accounts are healthy. No anomalies, no concerning communications.

## Step 1 — Health Anomaly Detection (`pipeline/health.py`)

This is the first filter. It runs on every account, costs nothing (pure pandas, no API calls), and produces a yes/no answer: does this account's usage pattern look anomalous?

### The five signals

**`usage_drop`** — Z-score of the last 7 days average API calls versus the stable baseline. If the recent average is more than 2 standard deviations below baseline, it flags. The z-score formula is: `(recent_mean - baseline_mean) / baseline_std`. A z-score of −2.0 means the recent usage is 2 standard deviations below what is normal for this account — statistically unusual.

**`wow_drop`** — Week-over-week comparison. If this week's average is more than 30% lower than last week's average, it flags. This catches sudden drops that might not show in the z-score yet (because the z-score uses a 60-day baseline, it is slower to react).

**`error_spike`** — Same z-score logic but applied to error rate. If the recent error rate is more than 2σ above baseline, the API is becoming unreliable for this customer.

**`webhook_drop`** — If the recent webhook success rate has dropped more than 5 percentage points below baseline, it flags. Webhooks failing means their product is breaking downstream.

**`rate_limit_stress`** — If there are 5 or more HTTP 429 responses in the last 7 days, the customer is hitting rate limits — either their usage is growing fast or something is wrong.

### The baseline contamination problem (and how it is solved)

This is the most subtle engineering decision in the health detector, and it is worth understanding carefully.

The baseline should represent "normal" behaviour for that account. If you include the anomalous period in the baseline, two things happen: the mean shifts down (making the recent drop look smaller), and the standard deviation goes up (making the z-score threshold harder to cross). Both effects reduce your ability to detect the anomaly.

The solution is `BASELINE_EXCLUDE = 30`. The baseline is computed from `df.iloc[:-30]` — all data **except** the last 30 days. For a 90-day dataset, this means the baseline covers days 0–59 cleanly. The planted anomalies start at day 76, so they never contaminate the baseline.

In production, you would recompute this daily (yesterday becomes part of the baseline, eventually). For a one-shot historical analysis like this, the fixed exclusion window is the right approach.

### Severity scoring

The severity level controls how much the correlator trusts this signal:

- `high`: usage drop paired with error spike or webhook drop (two systems failing simultaneously — a strong indicator of real disengagement, not just a slow week), or 3+ anomaly types at once.
- `medium`: exactly 2 anomaly types.
- `low`: exactly 1 anomaly type — could be noise, seasonal variation, or a transient issue.

The correlator uses severity to decide whether a weak LLM signal is enough to trigger outreach, or whether it needs strong LLM confirmation.

## Step 2 — LLM Risk Classification (`pipeline/classify.py`)

This step only runs on accounts that passed the health filter (i.e., `has_anomaly = True`). This is deliberate: LLM calls cost money and time. Running classification on all 100 accounts when only 30–40 have anomalies would be wasteful.

### What the LLM receives

The classifier assembles a context block from the three communication sources (tickets, Slack, transcripts) and passes it to the LLM with a carefully structured prompt. The prompt asks for a JSON response with five fields:

1. `use_case` — What are they actually building? (e.g., "real-time meeting transcription SaaS")
2. `churn_risk` — One of: low, medium, high
3. `churn_signals` — Specific quoted phrases from the communications that indicate risk
4. `recommended_action` — What the CS team should do (specific to this account)
5. `reasoning` — One sentence explaining the risk rating

Character limits are applied to each communication category to keep prompts within token budgets: 2000 chars for tickets, 1500 for Slack, 3000 for transcripts.

### Temperature and json_mode

The classification call uses `temperature=0.2` — very low, almost deterministic. This is intentional. You want consistent, structured output from the classification step. There is no value in creative variation here. The low temperature means that if you run the same account twice with the same communications, you get the same risk assessment.

`json_mode=True` tells the provider to constrain its output to valid JSON. For OpenAI this uses `response_format: {"type": "json_object"}`. For Anthropic there is no native JSON mode, so the system uses a technique called **assistant prefill**: the API call pre-fills the assistant's response with a `{` character, which strongly guides the model to complete it as a JSON object. The first character of the response is then re-attached when parsing.

### Robust parsing

The parser first tries `json.loads()`. If that fails (e.g., the model returned markdown-wrapped JSON), it falls back to a safe default: `churn_risk = "low"`, empty signals, generic recommendation. This means a bad LLM response never crashes the pipeline — it just results in the account not being triggered (conservative failure mode).

## Step 3 — Correlation Engine (`pipeline/correlate.py`)

This is the decision-making brain. It is pure Python — no ML, no LLM. It takes three inputs (account profile, health result, LLM classification) and applies business rules to produce a binary `should_trigger` decision plus a 0–1 confidence score.

### The rules, in order

**Free tier: never trigger.** Free accounts generate $0 in revenue. Customer Success time is finite. The correlator skips them entirely, regardless of how alarming the signals look.

**No anomaly: never trigger.** If the health detector found nothing, there is no evidence to act on. The LLM classifier was never even run for this account.

**Weak anomaly + not-high LLM risk: do not trigger.** A single signal (one of: wow_drop or error_spike alone) with a low or medium LLM risk assessment is probably noise. The seasonal FP traps pass through the health detector with a single `wow_drop` signal and get blocked here because the LLM correctly says "low risk." This is the critical rule that eliminates false positives.

**Multi-signal or high-confidence: trigger if LLM agrees.** If the health severity is medium or high, or there are multiple anomaly types, the correlator triggers outreach if the LLM says `churn_risk` is medium or high.

**Anomaly present but LLM says low risk: do not trigger.** The LLM's assessment of the communications takes precedence over the raw numbers when the LLM is confident the customer is healthy. This catches the "loyal competitor mention" FP trap.

### The confidence score

Confidence is a weighted composite: `0.40 × severity_weight + 0.40 × risk_weight + 0.20 × tier_weight`.

The weights encode business priorities: health signals and LLM risk are equally important (40% each), account value matters but is secondary (20%). An enterprise account with high anomaly severity and high LLM risk scores 1.0. A starter account with medium severity and medium risk scores about 0.5.

The confidence score is not used as a trigger threshold — the binary rules above decide whether to trigger. The confidence score is used by the dashboard to rank triggered accounts and by the evaluation notebook's precision-recall curve to show what a threshold-based system could achieve.

---

## Step 4 — Outreach Drafting (`pipeline/respond.py`)

If and only if `should_trigger = True`, the final step generates a draft outreach email. This email is not sent automatically — it is placed in the dashboard for a CS manager to review, edit, and send.

The prompt instructs the LLM to write as a customer success manager who speaks developer-to-developer. Key constraints in the prompt: address the specific technical issue detected (not generic "how are you"), reference the usage pattern or error rate specifically, offer one concrete next step, under 120 words, no marketing speak, no competitor names, no internal risk scores.

The `temperature` for this call is `0.5` — slightly higher than classification because natural language prose benefits from some variation. You do not want every email to sound like it was generated from the same template.

## The LLM Abstraction Layer (`llm/`)

The entire system is provider-agnostic. Every LLM call goes through `LLMClient.complete()`, an abstract base class. The concrete implementations (`OpenAIClient`, `AnthropicClient`) each implement `_complete_uncached()`. The factory reads the `LLM_PROVIDER` environment variable and instantiates the right client.

### Transparent disk caching

Every LLM call is cached automatically at the base class level, before any provider-specific code runs. The cache key is a SHA-256 hash of: provider name, model name, system prompt, user prompt, max_tokens, temperature, and json_mode. All of these must be identical for a cache hit.

This has two practical effects:

First, running the full pipeline twice costs money once. The second run is near-instant — all 76 LLM calls hit the disk cache.

Second, the dashboard's "Analyse & Draft" button is effectively free for accounts that have already been analysed with the same communications. The button only triggers a live API call when the communications checksum has changed.

The cache is stored in `cache/` (gitignored — thousands of small JSON files that are not useful in the repository).

---

## The Checksum Gate (`pipeline/utils.py` + dashboard)

The "Analyse & Draft" button in the dashboard is guarded by a communications checksum. When the pipeline runs, it computes a SHA-256 hash of the content of each account's tickets, Slack messages, and transcripts (sorted by ID for determinism, body + subject text only — timestamps and status fields are excluded).

This checksum is stored in `results.parquet` as `comms_checksum`.

When someone clicks "Analyse & Draft" in the dashboard:
1. The current checksum of the account's communications is recomputed from the raw JSON files.
2. It is compared to the stored checksum.
3. If they match and the account was already classified → show a green "Already analysed" notice. No API call.
4. If they differ or the account was never classified → show the button. On click, run the full classify → correlate → draft pipeline for that one account, write the result back to `results.parquet`, clear Streamlit's data cache, and reload.

Because the LLM calls themselves are also cached (by prompt hash), clicking the button on an already-analysed account even after a checksum mismatch will still be fast as long as the communications content is the same — only genuinely changed text triggers a new API call.

## The Dashboard (`dashboard/app.py`)

The Streamlit dashboard has three tabs.

### Tab 1 — Account List

A filterable table of all 100 accounts with four filters: tier, trigger status, churn risk, and anomaly severity. Triggered accounts (🔴) sort to the top. The table shows MRR, tier, anomaly severity, LLM risk, trigger decision, and confidence score.

This is the CS manager's morning view: which accounts need attention today?

### Tab 2 — Account Detail

Selecting any account shows:

- A **profile strip** with tier, MRR, funding stage, use case, anomaly severity, and trigger decision
- A **Plotly usage chart** showing the full 90-day API call volume with the anomaly window (last 14 days) highlighted in red on a dual y-axis with error count
- **Health signals** — a colour-coded list of all detected anomaly types plus a details table
- **LLM risk classification** — churn risk level, detected use case, and quoted churn signals from the communications
- **Trigger decision** — the correlator's reasoning in plain English
- **Draft outreach email** (if triggered) — displayed in a styled code block
- **"Analyse & Draft" button** — the checksum-gated re-analysis button described above

### Tab 3 — Evaluation

The evaluation tab loads `ground_truth.json` and compares it against the pipeline's `should_trigger` output. It shows: a confusion matrix heatmap, precision/recall/F1 metrics, per-signal-mix recall breakdown (health_only, risk_only, combined), and a per-account table of all planted churn cases and FP traps with their outcomes.

## Evaluation Results and What They Mean

Against the 100 synthetic accounts:

- **Precision: 100%** — Every account the system triggered for outreach was a genuine churn risk. Zero false positives.
- **Recall: 93%** — The system caught 14 of 15 planted churn cases. One was missed.
- **F1: 0.96** — Extremely high for a first-pass detection system.
- **FP traps blocked: 5/5** — All five false-positive traps were correctly not triggered.

The one miss is `acc_0009`, a risk_only account whose week-over-week usage decline was −21.8%. The detection threshold is 30%. The account passed through the health detector without an anomaly flag, so the LLM classifier was never called, so the correlator never saw it. In production, the threshold would be tuned on real labeled data — this is documented in the evaluation notebook's reflection section.

## Testing (`tests/`)

There are two test files:

**`tests/test_llm.py`** (12 passing, 2 skipped) — Tests the LLM abstraction layer. Verifies that a cache miss calls the provider, a cache hit skips it, different parameters produce different cache keys, and the factory raises `ValueError` on missing or unrecognised providers. Two live-API tests are skipped unless real API keys are present.

**`tests/test_pipeline.py`** (20 passing) — Tests the pipeline logic with no LLM calls and no file I/O. Three test classes:

- `TestHealthDetection`: builds a synthetic usage DataFrame in memory and verifies the health detector correctly identifies drops, leaves healthy accounts alone, and returns the expected output structure.
- `TestCorrelate`: directly exercises every branch of the correlator — free tier, no anomaly, weak-signal block, strong-signal trigger, low-risk block, enterprise boost.
- `TestChecksum`: verifies the checksum is stable, changes when content changes, is order-invariant, and ignores metadata fields.

## How to Talk About This Project

**"What is the problem you are solving?"**

Developer API customers often churn silently — they stop using the API, get frustrated by technical issues, explore competitors — but never file a cancellation ticket. By the time they churn, the revenue is already gone. This system detects the warning signs early by correlating usage health metrics with LLM-scored communications, and automatically drafts a personalised outreach email for the CS team.

**"Why not just use an LLM to analyse everything?"**

Three reasons. Cost: running LLM calls on every account every day would be expensive. The health detector acts as a cheap pre-filter, so the LLM only runs on the ~30–40% of accounts that have actually changed. Reliability: the statistics layer is deterministic and auditable — I can explain exactly why an account was flagged. Accuracy: the correlator's business rules act as a final gate so a hallucinated LLM response cannot trigger a false outreach.

**"How do you prevent false positives?"**

The correlator has a key rule: a single weak anomaly signal (e.g., one bad week) only triggers if the LLM **also** says high risk. This is the rule that blocks the seasonal dip and transient error FP traps. The seasonal trap gets a `wow_drop` flag but the LLM reads the communications, sees that the drop is explained by a planned holiday, and returns `churn_risk: low`. The correlator blocks it.

**"Why is it LLM-agnostic?"**

In production, you might want to switch providers based on cost, latency, or compliance requirements. The abstraction layer means the entire system works with one environment variable change (`LLM_PROVIDER=anthropic` or `openai`). The JSON mode implementation differs between providers — OpenAI uses a native parameter, Anthropic uses assistant prefill — but that difference is hidden inside each client class.

**"What would you do differently with real data?"**

The main limitations of synthetic data are: the LLM-generated communications are more clearly written than real developer messages (less abbreviation, ambiguity, sarcasm). Real data would need account-specific adaptive thresholds rather than fleet-wide z-score parameters. The feedback loop — CS marking each triggered account as saved or lost — would allow the classifier to improve over time. And real communications (from Slack, Pylon, Fireflies) would be considerably richer and noisier than synthetic text.
